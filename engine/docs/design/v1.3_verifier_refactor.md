# v1.3 Verifier Refactor: Two-Path Verification

Status: DESIGN. Not implemented. Awaiting Drew's approval before any code or
schema changes land.

Author: planner pass (Claude, Opus 4.7)
Created: 2026-05-18
Branch (suggested): `design/v1.3-verifier-refactor`
Supersedes (on approval): the single-path verifier described in
RUNNER_CONTRACT.md "Cold-read verification" (v1.2).

---

## 1. Problem

v1.2 ships a cold-read verifier as a reasoning agent invocation per card,
skippable on a high-confidence cascade-clean run. Per-card verifier cost is
roughly 5 to 15k tokens (Sonnet plus extended thinking, or Opus). Skip rate
is somewhere in the 50 to 70 percent range, leaving an average verification
cost of 1.5 to 5k tokens per card.

The schema already mandates machine-checkable AC items
(`acceptance_checks:` with a `check.type` field). If the contract is
honest, most checks need zero LLM tokens to verify. A deterministic runner
can execute them. The reasoning agent only earns its keep when an item is
explicitly marked subjective (already permitted on tier 5 and 6 cards under
v1.1).

v1.3 splits the verifier into two paths:

1. Deterministic runner. Free, fast, no LLM call.
2. Subjective evaluator. Single Haiku-class call, batched across all
   subjective items on the card, only invoked if the card has any.

Common case (cards with only deterministic AC) drops to zero verifier
tokens. Outlier case (tier 5 or 6 with subjective items) drops from a
Sonnet-plus-ET pass to a single Haiku batch.

## 2. AC item structured schema

### 2.1 Canonical types

Each AC item declares a `type`. The runner dispatches on the type to the
matching runner-side implementation. Types fall into three families:
filesystem, process, network, and human-judgment.

**Filesystem family.**

- `file_exists`. Assert a path exists. Required: `path` (string,
  worktree-relative or absolute). Pass if `os.path.exists(path)` is true.
- `file_absent`. Inverse of `file_exists`. Useful for "the migration script
  was deleted" checks.
- `file_contains`. Assert a file contains a pattern. Required: `path`
  (string), plus exactly one of `pattern` (regex, Python `re` flavor) or
  `literal` (exact substring). Optional: `case_sensitive` (default true).
  Pass if at least one match is found.
- `file_absent_content`. Assert a file does NOT contain a pattern. Same
  shape as `file_contains` with inverted pass condition. Use for "the
  hardcoded API key was removed" style checks.

**Process family.**

- `command`. Execute a shell command. Required: `command` (string).
  Optional: `expected_exit_code` (int, default 0), `cwd` (string, default
  worktree root), `env` (map of overrides, default empty), `timeout_sec`
  (int, default 60). Capture stdout and stderr to `verifier_notes` on
  fail. Pass if `exit_code == expected_exit_code`.
- `python_assert`. Evaluate a Python expression that must return truthy.
  Required: `expression` (string). Optional: `imports` (list of module
  names to make available, default `["os", "json", "pathlib", "re"]`).
  Pass if `bool(eval(expression))` is true. Open question 5 covers the
  available namespace.

**Network family.** Only enabled when the project config opts in. Carry
over the existing `network_checks_allowed` gate.

- `http_status`. Make an HTTP request, assert status. Required: `url`
  (string), `expected_status` (int or list of ints). Optional: `method`
  (default GET), `headers` (map), `body` (string or JSON), `timeout_sec`
  (default 10), `retries` (default 0, see open question 3).
- `http_contains`. Make an HTTP request, assert body matches a pattern.
  Same request fields as `http_status` plus `pattern` or `literal` (one of)
  and `case_sensitive` (default true).

**Human-judgment family.**

- `subjective`. A Haiku-class LLM evaluates this item against the
  executor's claimed evidence. Required: `description` (already present),
  `evidence_required` (string, what the executor must produce in
  completion notes for the evaluator to read). Reserved for tier 5 and 6
  cards. Validator MUST refuse a batch where a tier 1 to 4 card carries a
  subjective item.

### 2.2 Other types considered, not included

- `json_path`. Assert a JSON document's path equals a value. Probably
  useful, but `python_assert` covers the cases until we see demand.
- `sql_query`. Assert a query against a project DB returns N rows or a
  specific value. Out of scope until a card actually needs it. Same
  reasoning as `json_path`.
- `image_diff`. Visual regression. Reasonable in a frontend repo. Defer
  to a future plugin-style type.

Recommendation: ship the seven types above (four filesystem, two process,
two network, one subjective) and defer the rest until a card forces the
question.

### 2.3 Naming reconciliation with v1.2

The v1.2 template uses `acceptance_checks:` with type names `shell`,
`file_exists`, `file_absent`, `grep_match`, `grep_absent`, `http_status`.
The user-supplied v1.3 spec uses `acceptance_criteria:` and renames
`shell` to `command` and `grep_match` to `file_contains`.

Options-with-reasoning:

- A. Adopt the v1.3 names verbatim. Cleaner, but every existing card and
  the template need to migrate. Mechanical, but real work.
- B. Keep the v1.2 names (`acceptance_checks`, `shell`, `grep_match`)
  and add the new types alongside. Zero migration cost, but the
  vocabulary stays inconsistent forever.
- C. Adopt v1.3 names AND accept v1.2 names as deprecated aliases until
  v1.4. Best of both, costs the runner a tiny alias table.

Recommend C. The alias table lives in the runner library and is the
shortest deprecation path that doesn't blow up live work.

## 3. Card template update

Excerpt of the AC block after v1.3 (option C naming):

```yaml
acceptance_criteria:
  # Filesystem.
  - description: "rate limit middleware exists"
    type: file_exists
    path: "src/middleware/rate_limit.ts"

  - description: "no hardcoded secrets remain in config"
    type: file_absent_content
    path: "src/config/default.ts"
    pattern: "(api_key|secret_key|password)\\s*[:=]\\s*['\"][^'\"]+['\"]"

  # Process.
  - description: "rate limit tests pass"
    type: command
    command: "npm test -- --grep 'rate limit'"
    expected_exit_code: 0
    timeout_sec: 120

  - description: "lint clean"
    type: command
    command: "npm run lint"

  - description: "rate limit headers expose retry-after"
    type: python_assert
    expression: "'Retry-After' in open('src/middleware/rate_limit.ts').read()"

  # Network. Project must opt in via network_checks_allowed: true.
  - description: "health endpoint returns 200"
    type: http_status
    url: "http://localhost:8080/health"
    expected_status: 200
    retries: 3

  # Subjective (tier 5 / 6 only).
  - description: "error response is well-formed"
    type: subjective
    evidence_required: >
      The executor must paste at least one rendered error response from
      the new middleware in the completion notes. The evaluator judges
      whether the message is clear, actionable, and matches the
      project's documented error voice (see docs/style/errors.md).
```

Top-level YAML frontmatter changes:

- `acceptance_criteria:` replaces `acceptance_checks:` (with deprecation
  alias).
- Per-item `subjective: true` flag (v1.1) is replaced by `type:
  subjective`. Same intent, cleaner shape.
- New optional top-level field: `verifier_schema_version: "1.3"`. Cards
  without this are treated as v1.2 and routed through the
  backward-compat path (section 7).

Nothing else in the frontmatter changes.

## 4. Verifier orchestrator design

Pseudocode for the v1.3 verifier flow. Lives in the runner repo
(see open question 1).

```python
def verify_card(card_path: Path, worktree: Path,
                project_cfg: ProjectConfig) -> VerifierResult:
    card = parse_card(card_path)
    ac_items = card.frontmatter["acceptance_criteria"]
    results: list[ItemResult] = []
    subjective_items: list[tuple[int, ACItem]] = []

    # Pass 1: dispatch deterministic items.
    for idx, item in enumerate(ac_items):
        t = item["type"]
        if t == "subjective":
            subjective_items.append((idx, item))
            continue
        runner_fn = DETERMINISTIC_RUNNERS[t]   # KeyError = bad schema
        result = runner_fn(item, worktree, project_cfg)
        results.append(ItemResult(idx, item, result))

    # Pass 2: batch subjective items in one Haiku call (only if any).
    if subjective_items:
        evidence = card.completion_notes or extract_executor_evidence(card)
        subj_results = evaluate_subjective_batch(
            items=[i for _, i in subjective_items],
            card_body=card.body,
            evidence=evidence,
            model="claude-haiku-4-5-20251001",
        )
        for (idx, item), res in zip(subjective_items, subj_results):
            results.append(ItemResult(idx, item, res))

    # Sort by original index for stable output.
    results.sort(key=lambda r: r.idx)

    if all(r.passed for r in results):
        return VerifierResult.pass_(results)
    return VerifierResult.fail_(results, notes=build_notes(results))
```

`evaluate_subjective_batch` makes ONE Haiku call with all subjective
items, the card body, and the executor's claimed evidence. The response
is a structured list with pass/fail and brief reasoning per item. One
call regardless of how many subjective items the card has. Tier 6 cards
that pile up subjective items still cost one Haiku call.

Outcomes hook into the existing state machine unchanged:

- All pass: stamp `verified_at`, `verified_by`, move card to `done/`.
- Any fail: append structured `verifier_notes:` to card body, return card
  to `active/`, clear `claimed_by` / `started_at` / `last_heartbeat`.
- Runner crash or Haiku error: existing two-retry policy, then
  `blocked/`.

The cascade-on-confidence skip path stays, but it only skips the
deterministic pass. Open question: should it also skip the subjective
pass on tier 5 / 6 cards? Recommendation: no. Subjective items on
high-stakes cards are exactly the spots where humans expect a second
look, even when the executor was confident. Cheap insurance at Haiku
prices.

### 4.1 Runner-side implementation surface

```
DETERMINISTIC_RUNNERS = {
    "file_exists":         run_file_exists,
    "file_absent":         run_file_absent,
    "file_contains":       run_file_contains,
    "file_absent_content": run_file_absent_content,
    "command":             run_command,
    "python_assert":       run_python_assert,
    "http_status":         run_http_status,
    "http_contains":       run_http_contains,

    # Deprecated v1.2 aliases (warn on use, remove in v1.4).
    "shell":               run_command,
    "grep_match":          run_file_contains,
    "grep_absent":         run_file_absent_content,
}
```

Each runner function is a pure(-ish) function taking the AC item dict,
the worktree path, and the project config, returning a
`(passed: bool, evidence: dict)` tuple. Evidence is whatever the
verifier writes into `verifier_notes` on fail: stdout/stderr for
commands, the missing path for file checks, the HTTP body snippet for
network checks, etc.

## 5. Runner contract changes

The "Cold-read verification" section of `RUNNER_CONTRACT.md` gets
rewritten end-to-end. Diff summary:

### 5.1 Two-path model

New top-level subsection. States that verification consists of:

- A deterministic phase that runs every non-subjective AC item via the
  runner library. Zero LLM tokens.
- A subjective phase that runs at most once per card, only if the card
  has `type: subjective` items, as a single batched Haiku call.

Removes the "verifier agent is invoked per card" framing.

### 5.2 Runner library requirement

New normative paragraph. The runner MUST provide an implementation for
every type in section 2.1. The implementation surface is the
`DETERMINISTIC_RUNNERS` table in section 4.1. Runners MAY add custom
types, but custom types MUST be namespaced (`project:foo`) so the
validator can distinguish core from project extensions.

Cards that reference a type the runner doesn't implement land in
`blocked/` with a clear error, not silently passed.

### 5.3 Security and trust boundary

`command` and `python_assert` items can execute arbitrary code. This is
not new (v1.2 `shell` does the same), but v1.3 is a good moment to
state the trust model explicitly. Options-with-reasoning:

- A. Trust the card author. Cards are planner output, the planner is a
   trusted agent in our threat model, and the same code runs during
   executor work anyway. Status quo. Cheapest. Risk: a compromised
   planner or a malicious manifest can run anything.
- B. Sandbox via docker / nsjail / bubblewrap. The verifier runs each
   `command` and `python_assert` inside a constrained namespace with
   read-only access to the worktree and no network unless the project
   opts in. Strong isolation. Real ops cost: docker has to be present,
   image needs maintenance, cold-start latency adds up over a 50-card
   sprint.
- C. Worktree-isolated process, no extra sandbox. The verifier runs the
   check in a fresh shell rooted at the worktree, with `HOME` and `PATH`
   scrubbed to a minimal set. No docker, just hygiene. Mid-strength.
   Catches accidents and casual mistakes, doesn't stop a determined
   attacker.
- D. Allow-list of binaries for `command`. The project config names the
   binaries verifier commands may invoke (`npm`, `pytest`, `make`, etc.).
   Anything else fails closed. Predictable. Annoying when a new tool
   shows up and the allow-list isn't updated.

Recommend C as the baseline, with B as an opt-in for projects that
want it (`verifier_sandbox: docker` in project config). Pair with a
project-level `verifier_command_allow_list:` (option D) as opt-in for
the paranoid. A is the de facto current state and would let us ship
v1.3 without ops work, but we should not advertise it as "secure"; if
we pick A, the docs should say so plainly.

### 5.4 Skip rule update

The "verifier MAY be skipped" subsection narrows. The skip applies only
to the deterministic phase (which is already free, so the skip is
mostly bookkeeping). Subjective items, if any, always run unless the
card is in `blocked/`. The three preconditions (cascade-clean,
confidence threshold, AC clean first run) still apply.

The `verifier_skipped_reason` field semantics tighten: a card with any
subjective items can never be fully skipped, so the field becomes
`verifier_skipped_reason: "deterministic phase only; subjective items
ran"` in mixed cases.

### 5.5 Manual override

`run_verifier(card_id)` semantics unchanged. Output schema gains a
`phase` field per item result: `"deterministic"` or `"subjective"`.

## 6. Token cost analysis

Numbers are estimates from the v1.2 handoff + the user's framing. Treat
as order-of-magnitude until we have telemetry.

### 6.1 Pre-v1.3 (current)

- Per-card verifier when invoked: 5 to 15k tokens. Sonnet + ET, or Opus
  on tier 5 / 6.
- Skip rate via cascade-clean: 50 to 70 percent.
- Average per-card verification cost: roughly 1.5 to 5k tokens.

### 6.2 Post-v1.3

- Deterministic items: zero LLM tokens.
- Subjective items only on tier 5 / 6 cards that include them. Realistic
  fraction of cards in a disciplined planner's batch: 5 to 15 percent.
- When invoked, Haiku batch cost: roughly 1 to 3k tokens. Card body plus
  evidence is the bulk; the items themselves are short.
- Average per-card verification cost: roughly 100 to 500 tokens
  (weighted: 85 to 95 percent of cards at zero, 5 to 15 percent at 1 to
  3k).

### 6.3 Sprint-scale

For a 50-card sprint:

| Path  | Avg tokens / card | Sprint total      |
|-------|-------------------|-------------------|
| v1.2  | 1.5k to 5k        | 75k to 250k       |
| v1.3  | 100 to 500        | 5k to 25k         |
| Saved |                   | ~70k to ~225k     |

At current Sonnet pricing the savings translate to a couple of dollars
per sprint per project. Modest in raw dollars, real as a fraction of
verification cost (roughly 90 to 95 percent reduction). The latency win
is bigger than the dollar win: deterministic checks finish in seconds,
not the 30 to 90 seconds an LLM round-trip takes.

Second-order win: less LLM-in-the-loop means fewer flaky verifier
outcomes from the model wandering off, fewer retries, less drift in
verifier_notes wording over time.

## 7. Backward compatibility

The user's framing assumes v1.2 cards have AC items as bullet strings.
The actual v1.2 template already has structured `acceptance_checks:`.
The compat surface is therefore smaller than feared.

Two real compat axes:

A. **Field rename** (`acceptance_checks` -> `acceptance_criteria`,
   `shell` -> `command`, `grep_match` -> `file_contains`,
   `grep_absent` -> `file_absent_content`).
B. **Subjective flag** (`subjective: true` -> `type: subjective`).

Options-with-reasoning:

- 1. Auto-migrate on next read. The runner reads any v1.2 card, rewrites
     the AC block in-memory under the v1.3 schema using the alias
     table, and (optionally) writes the rewritten card back to disk on
     the next normal mutation (status change, heartbeat, whatever). Low
     friction. Risk: silent on-disk drift if a human has the card open
     in an editor.
- 2. Hard deprecation. v1.3 runner refuses to verify v1.2-schema cards
     and flags them for manual update. Clean, but breaks any in-flight
     work the moment the runner upgrades.
- 3. Alias-with-warning. Runner accepts v1.2 names indefinitely (or
     until v1.4) via the alias table in section 4.1, emits a deprecation
     warning to the runner log per card, and does not rewrite disk.
     Existing cards keep working, new cards use v1.3 names, the planner
     stops emitting v1.2 names on the v1.3 upgrade.
- 4. Combination: alias-with-warning until end of v1.3, hard deprecation
     in v1.4. Migration script (`/cards migrate-ac-schema`) provided to
     do the rewrite explicitly when a human is ready.

Recommend 4. Matches the project's general preference for explicit,
auditable migrations over silent rewrites. The migration script is
small (regex + YAML rewrite) and gives Drew a single command to run
once.

Special case: a v1.2 card with `subjective: true` and no `type:` on the
item. Under v1.3 alias resolution, the runner treats it as `type:
subjective` and ignores any other check-shape fields. The deprecation
warning fires.

## 8. Open design questions

Not answered here. These need Drew's call before implementation.

1. **Runner library location.** Lives in `agile-cards` (so the contract
   and the implementation ship together) or in the runner repo (so the
   contract stays an interface and any conforming runner can implement
   it)? Tradeoff: portability vs. drift.

2. **Working directory and env for `command` items.** Implicit (always
   worktree root, project-config env) or explicit per card (every
   `command` item must carry `cwd` and `env`, even if blank)? Implicit
   is shorter cards. Explicit is no surprises.

3. **HTTP retry policy.** Should `http_status` and `http_contains`
   default to retrying flaky network checks? If yes, default count and
   backoff? Suggested defaults if we go with retries: 3 attempts,
   exponential backoff starting at 1 second.

4. **Default timeouts.** Per-type defaults:
   - `command`: 60 sec? 300 sec? Both are common.
   - `python_assert`: 5 sec.
   - `http_status` / `http_contains`: 10 sec.
   - Filesystem types: no timeout (synchronous syscall).
   Project config should be able to override.

5. **`python_assert` namespace.** Stdlib only (`os`, `json`, `pathlib`,
   `re`, `sys`, `subprocess`)? Project-specific bindings (e.g., an
   imported `project_helpers` module the team maintains)? Locked-down
   subset (no `subprocess`, no `open` for write)? Pairs with the
   security discussion in section 5.3.

6. **Subjective evaluator model pinning.** v1.3 says "Haiku-class." The
   exact model string (`claude-haiku-4-5-20251001` as of today) goes in
   project config or is runner-pinned? Pinning in project config lets
   teams stay on a known-good model; pinning in the runner makes
   upgrades centralised.

7. **Evidence contract for subjective items.** Should the executor be
   required to write evidence in a structured block in completion notes
   (a `subjective_evidence:` YAML block), or is free-form prose enough?
   Structured is easier for the evaluator. Free-form is what executors
   do naturally.

8. **What happens when an AC item has an invalid `type`.** Hard fail at
   verification time, or planner-side validation that refuses the batch?
   Recommend both, but state it explicitly.

9. **Should `verifier_schema_version: "1.3"` be required on new cards,
   or implicit?** Required is auditable. Implicit means the planner
   doesn't have to add it. Probably required given how the project
   treats schema discipline elsewhere.
